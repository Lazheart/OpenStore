Deployment

CloudFormation, Docker y despliegue reproducible

La infraestructura está declarada en CloudFormation para poder recrearse de forma determinista. Los servicios se empaquetan con Docker, las bases se levantan con Docker Compose y el frontend se publica por Amplify.

Por qué CloudFormation

Infraestructura como código, no como clics manuales

CloudFormation deja la infraestructura declarada en un template versionable. Eso evita cambios manuales difíciles de rastrear, permite repetir la misma topología en otra cuenta AWS y hace más simple revisar qué recursos necesita realmente el sistema.

Docker por servicio

Microservicios

Cada servicio del backend tiene su propio Dockerfile, permitiendo construir imágenes independientes y desplegarlas de forma aislada sin afectar a los demás.

CI/CD automático

El workflow de GitHub Actions (.github/workflows/docker-image.yml) construye y publica las imágenes automáticamente en cada push a la rama configurada.

Docker Compose — entorno local

database/docker-compose.yml

Levanta PostgreSQL en 5432, MySQL en 3307 y MongoDB en 27017. Cada contenedor conserva su volumen para no perder datos entre reinicios.

backend/docker-compose.yml

Arranca user-service, shop-service, product-service y store-service con healthchecks y variables de entorno predefinidas.

Frontend en Amplify

Build

La carpeta frontend/ se compila con Vite usando pnpm install y pnpm run build. El resultado sale en dist/.

Amplify

CloudFormation crea la app de Amplify apuntando al repositorio. El frontend se despliega automáticamente desde la rama configurada sin tocar el servidor.

Buenas prácticas

Usa un rol dedicado

Crea un IAM Role específico para ejecutar la plantilla en lugar de la cuenta raíz. Reduces superficie de error y delimitas qué acciones pueden realizar las pruebas.

Aplica least privilege

Limita permisos a EC2, ELB, S3, CloudFormation y Amplify estrictamente a lo necesario para el despliegue. Menos permisos equivale a menos riesgo.

Importante: usa un rol IAM y un usuario específico para el despliegue en lugar de la cuenta raíz. Limita permisos a lo estrictamente necesario (EC2, ELB, S3, CloudFormation, Amplify) y aplica least privilege. El objetivo es reducir la superficie de riesgo y limitar el impacto si algo falla.
Con cuentas limitadas se recomienda usar el rol proporcionado por la institución educativa para el despliegue. Aunque esto sea una vulnerabilidad de seguridad.
